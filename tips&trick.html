What is variable hoisting? The JavaScript engine treats all variable declarations using ‚Äúvar‚Äù as if they are declared at the top of a functional scope(if declared inside a function) or global scope(if declared outside of a function) regardless of where the actual declaration occurs. This essentially is ‚Äúhoisting‚Äù. So variables might actually be available before their declaration. Let‚Äôs see it in action.. // OUTPUT : undefined console.log(shape); var shape = "square"; // OUTPUT : "square" console.log(shape); If you‚Äôre coming from C-based languages, you‚Äôd expect an error to be thrown when the first console.log is called since the variable shape hadn‚Äôt been defined at that point. But the JavaScript interpreter looks ahead and ‚Äúhoists‚Äù all variable declarations to the top, and the initialization remains in the same spot. Here‚Äôs what is happening behind the scenes: //declaration getting hoisted at the top var shape; // OUTPUT : undefined console.log(shape); var shape = "square"; // OUTPUT : "square" console.log(shape); Here is another example this time in a functional scope to make things more clear: function getShape(condition) { // shape exists here with a value of undefined // OUTPUT : undefined console.log(shape); if (condition) { var shape = "blue"; // some other code return shape; } else { // shape exists here with a value of undefined return false; } } You can see in the above example how shape‚Äôs declaration is hoisted at the top of getShape() function. It is therefore accessible everywhere outside the if block and within the function with an undefined value. This default behavior of JavaScript has its fair share of advantages and disadvantages. Not fully understanding these can leads to bugs in our code. Enter Block-Level Declarations! ES6 introduced block-level scoping options to provide developers with more control and flexibility over a variable‚Äôs lifecycle. Block-level declarations are made in block/lexical scopes that are created inside a block ‚Äú{ }‚Äù. let Declarations This syntax is similar to var, just replace var with let to declare a variable with its scope being only that code block. Place your let declarations at the top in a block so they‚Äôll be available within that entire block. For example: function getShape(condition) { // shape doesn't exist here // console.log(shape); => ReferenceError: shape is not defined if (condition) { let shape = "blue"; // some other code return shape; } else { // shape doesn't exist here return false; } } Notice how shape exists only inside the if block, and throws an error when accessed outside of it instead of outputting undefined like our previous case with var declarations. NOTE : If an identifier has already been defined inside a scope , using that same identifier in a let declaration inside the same scope throws an error. Also, no error is thrown if a let declaration creates a variable with the same name as that of a variable in it‚Äôs outer scope. (This case is same with const declarations which we‚Äôll talk about shortly.) For example: var shape = "square"; let shape = "rectangle"; // SyntaxError: Identifier 'shape' has already been declared and: var shape = "square"; if (condition) { // doesn't throw an error let shape = "rectangle"; // more code } // No error const Declarations The declaration syntax is similar to let & var , lifecycle is the same as let. But you have to follow some rules. Bindings declared using const are treated as constants, and therefore their values cannot be changed once defined. Due to this, every const declaration must be initialized at the time of declaration. For example: // valid const shape = "triangle"; // syntax error: missing initialization const color; // TypeError: Assignment to constant variable shape = "square" However, the rule is a little different when it comes to objects. An object‚Äôs value can be modified! const shape = { name: "triangle", sides: 3 } // WORKS shape.name = "square"; shape.sides = 4; // SyntaxError: Invalid shorthand property initializer shape = { name: "hexagon", sides: 6 } In the above example we can see that only the properties of the shape object could be changed because we are only changing what shape contains, not what it‚Äôs bound to. We can summarize by saying that const prevents modification of the binding as a whole‚Ää‚Äî‚Äänot the value that it‚Äôs bound to. Note: properties can be mutated. So for true immutability, use Immutable.js or Mori. The Temporal Dead Zone We now know that accessing a let or const variables before they‚Äôre declared will throw a ReferenceError. This period between entering scope and being declared where they cannot be accessed is called the Temporal Dead Zone. Note that ‚ÄúTemporal Dead Zone‚Äù isn‚Äôt formally mentioned in the ECMAScript specification. It‚Äôs just a popular term among programmers. I personally recommend you always use let or const, as they lead to fewer bugs. I have yet to encounter a situation where I needed to use var. As a general rule, I use let for loop counters or only if I really need reassignment. Everywhere else, I use const. How we‚Äôre building a platform for future of eCommerce

In order to launch an online store in Iran, you have to hire devs or an agency to build you a site. There is no Shopify or anything close to that (in terms of service and ease of use). You have to spend a lot of money to just launch a site. Not to mention, there are lots of rules and regulation about how your site should work. Let‚Äôs get a bit technical Kamva eCommerce platform is consist of three different apps, Shop Admin, Shop front, and API. We designed it to be really easy to use. The shop admin app and shop front app is written in Backbone.js. They are single page application (SPA) which have two important advantages. This will allow us to launch our mobile apps and integrate with other services quicker. It also allow us to expand to other markets fast. One thing was really important for us in Kamva, we wanted to build a platform that helps our customers sell more. For that reason, we are giving away Elasticsearch on every plan.
